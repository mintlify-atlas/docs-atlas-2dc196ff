TDgpt is an intelligent analytics agent built into TDengine that lets you run advanced time-series analysis directly from SQL. You can invoke statistical algorithms, machine learning models, deep learning models, time-series foundation models, and large language models — all through SQL statements, against your data in TDengine.

TDgpt exposes its capabilities through two new SQL constructs:

- **`FORECAST()`** — predict future values based on historical time-series data
- **`ANOMALY_WINDOW()`** — detect anomalies and return the time windows in which they occurred

## Architecture

TDgpt is composed of one or more stateless **AI nodes (anodes)**. Anodes can be deployed across the TDengine cluster on hardware appropriate for the algorithms being used — including GPU-equipped machines for deep learning workloads.

During query execution, the vnode in TDengine forwards any advanced analytics elements to the anode. Once analysis is complete, results are assembled and embedded back into the query execution flow.

TDgpt includes four main components:

- **Built-in analytics libraries**: statsmodels, pyculiarity, pmdarima — ready-to-use models for forecasting and anomaly detection.
- **Built-in ML libraries**: PyTorch, Keras, Scikit-learn — for running pre-trained machine and deep learning models.
- **Adapter for locally deployed time-series models**: Connects to Time-MoE and TDtsfm, which are purpose-built for time-series data and are lighter-weight than general-purpose LLMs.
- **Adapter for cloud time-series MaaS**: Connects to cloud-hosted services such as TimeGPT.

Two Docker images are available:

  <Step title="Pull the image">

    ```bash
    docker pull tdengine/tdgpt-full:latest
    ```
  </Step>
  <Step title="Start the container">
    ```bash
    docker run -d \
      -p 6035:6035 \
      -p 6061:6061 \
    ```

    For the full image (also exposes the Time-MoE model service on port 6062):
    ```bash
    docker run -d -p 6035:6035 -p 6061:6061 -p 6062:6062 tdengine/tdgpt-full:latest
    ```

    Port reference:
    - `6035`: TDgpt REST service
    - `6061`: TDtsfm model service
    - `6062`: Time-MoE model service
  </Step>
  <Step title="Confirm the container is running">
    ```bash
    docker ps
    ```
  </Step>
</Steps>

## Time-Series Forecasting with FORECAST()

The `FORECAST()` function takes historical time-series data as input and returns predicted future values.

### Syntax

```sql
FORECAST(column_expr, option_expr)

option_expr: {"
  algo=expr1
  [,wncheck=1|0]
  [,conf=conf_val]
  [,every=every_val]
  [,rows=rows_val]
  [,start=start_ts_val]
  [,expr2]
"}
```

### Parameters

| Parameter | Description | Default |
|-----------|-------------|---------|
| `algo` | Forecasting algorithm to use | `holtwinters` |
| `wncheck` | White noise check: `1` to enable, `0` to disable | `1` |
| `conf` | Confidence interval (0–100) | `95` |
| `every` | Sampling period for forecast output | Sampling period of input data |
| `start` | Starting timestamp for forecast data | One period after the last input timestamp |
| `rows` | Number of forecast rows to return (max 1024) | `10` |

### Pseudocolumns

Three pseudocolumns are available in forecast results:

| Pseudocolumn | Description |
|-------------|-------------|
| `_FROWTS` | Timestamp of the forecast data point |
| `_FLOW` | Lower bound of the confidence interval |
| `_FHIGH` | Upper bound of the confidence interval |

### Examples

```sql
-- ARIMA forecast: 10 rows (default), white noise check enabled, 95% confidence interval
SELECT _flow, _fhigh, _frowts, FORECAST(i32, "algo=arima")
FROM foo;

-- ARIMA forecast with periodic data: 10 samples per period, white noise check disabled
SELECT _flow, _fhigh, _frowts, FORECAST(i32, "algo=arima,alpha=95,period=10,wncheck=0")
FROM foo;
```

Sample output:

```

========================================================================================

          -21.9861946 |           83.3938904 | 2020-01-01 00:01:36.000 |            30 |
```

## Anomaly Detection with ANOMALY\_WINDOW()

`ANOMALY_WINDOW` is a new type of window clause that identifies time ranges containing anomalies. You use it in a `WHERE` clause, and aggregate queries against the detected anomaly windows.

### Syntax

```sql
ANOMALY_WINDOW(column_name, option_expr)

option_expr: {"
  algo=expr1
  [,wncheck=1|0]
  [,expr2]
"}
```

| Parameter | Description | Default |
|-----------|-------------|---------|
| `algo` | Anomaly detection algorithm | `iqr` |
| `wncheck` | White noise check: `1` to enable, `0` to disable | `1` |

Three pseudocolumns describe each anomaly window:

| Pseudocolumn | Description |
|-------------|-------------|
| `_WSTART` | Start timestamp of the anomaly window |
| `_WEND` | End timestamp of the anomaly window |
| `_WDURATION` | Duration of the anomaly window |

### Examples

```sql
-- Detect anomalies in i32 using the default IQR algorithm.
-- Return the anomaly window boundaries and the sum of i32 within it.
SELECT _wstart, _wend, SUM(i32)
FROM foo
ANOMALY_WINDOW(i32, "algo=iqr");

-- Use the k-sigma algorithm with k=2.
SELECT _wstart, _wend, SUM(i32)
FROM foo
ANOMALY_WINDOW(i32, "algo=ksigma,k=2");
```

Sample output:

```
         _wstart         |          _wend          |   count(*)    |
====================================================================
 2020-01-01 00:00:16.000 | 2020-01-01 00:00:17.000 |             2 |
```

You can also use `ANOMALY_WINDOW` as the inner query in a nested query, with all the same aggregation functions available for other windowed queries.

### Built-In Anomaly Detection Algorithms

| Category | Algorithm | Description |
|----------|-----------|-------------|
| Statistical | `iqr` | Interquartile range (default) |
| Statistical | `ksigma` | K-sigma outlier detection |
| Statistical | `grubbs` | Grubbs test |
| Density-based | `lof` | Local Outlier Factor |
| Seasonal | `shesd` | Seasonal ESD |
| ML | `sample_ad_model` | Autoencoder-based model |

## Managing Anodes

### Refresh Algorithm Cache

After adding a custom algorithm or updating models, refresh the algorithm cache:

```sql
UPDATE ANODE 1;
-- or refresh all anodes at once:
UPDATE ALL ANODES;
```

### Remove an Anode

```sql
DROP ANODE 1;
```

<Note>
Dropping an anode only removes it from the TDengine cluster registry. To stop the anode process, use `systemctl stop taosanoded` on the anode machine. To fully uninstall it, run `rmtaosanode`.
</Note>

## Custom Algorithms

TDgpt is an open platform. You can add your own forecasting, anomaly detection, imputation, or classification algorithms in Python. Once added, you reference them by name in SQL — no application code changes required.

To add a custom algorithm:

1. Implement the algorithm following TDgpt's interface requirements.
2. Place the source files in the appropriate anode directory and restart the anode.
3. Refresh the algorithm cache with `UPDATE ANODE` or `UPDATE ALL ANODES`.

You can then use your algorithm by setting `algo=your_algorithm_name` in any `FORECAST()` or `ANOMALY_WINDOW()` call.
